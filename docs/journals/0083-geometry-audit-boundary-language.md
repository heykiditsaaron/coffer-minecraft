# 0083 Geometry Audit: Boundary Language

## Summary

Audit the current high-level docs and recent journals under the bounded-lawful
reality-layer lens.

This is a wording audit and boundary map, not a schema change and not a
doctrinal rewrite.

## Files Reviewed

- `README.md`
- `docs/architecture/repository-boundaries.md`
- `docs/contracts/README.md`
- `docs/integration/README.md`
- `docs/journals/0077-contact-topology-review.md`
- `docs/journals/0081-boundary-classification-review.md`
- `docs/journals/0082-boundary-evaluation-checklist.md`

## What Remains Aligned

- The repository-level docs still correctly separate Minecraft-specific work
  from substrate ownership.
- The journals still distinguish contact topology from schema expansion.
- The checklist already frames future seam work as a contact/earnings question
  rather than a content-expansion assumption.
- The boundary classification journal already keeps mutation refusal in the
  container-boundary category and omission discipline in the Coffer-intrinsic
  category.

## What May Need Later Wording Clarification

- "Contact" vs "interpretation" still benefits from explicit reminders that
  not every observed result is a boundary contact.
- "Participation" vs "outcome" may need more precision when a boundary is
  reached but returns a non-success result.
- "Absence" vs "omission-as-content" may need careful wording so missing fields
  are not mistaken for accidental silence.
- "Evidence" vs "explanation" may need explicit separation in future journals
  that discuss why a boundary behaved the way it did.

## Future Pressure Points

- Any future seam proposal that wants to add earned detail should first pass
  the checklist questions in `0082`.
- Mutation remains the main pressure point because it currently sits at the
  container-boundary refusal layer.
- Runtime wording may need additional care if future work introduces a clearer
  distinction between non-success contact and refusal.
- Core/Runtime/Authority/Adapter responsibilities should continue to be stated
  as ownership boundaries, not as evidence of a final ontology.

## What Must Not Change Yet

- No code changes.
- No schema changes.
- No SER/CER envelope changes.
- No mutation seam forcing.
- No gameplay, commands, or UX expansion.
- No rewrite of Core Law or final doctrine claims.
- No attempt to turn these current best-fit interpretations into permanent
  metaphysical truth.

## Rationale

The current docs are mostly aligned with the bounded-lawful reality-layer lens.
The remaining risk is not structural conflict, but wording drift: future notes
could blur whether a statement is about actual contact, a lawful refusal,
inference about contact, or a narrative explanation of why contact behaved the
way it did.

## Verification

No implementation changed for this audit.

## Uncertainties

The strongest remaining ambiguity is how future notes should phrase
non-success contact results so they stay clearly separate from both refusal and
successful participation.
